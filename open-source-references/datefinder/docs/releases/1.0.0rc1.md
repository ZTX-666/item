# datefinder 1.0.0rc1

Release date: 2026-03-19

## Highlights

- New v2 extraction engine is now the default for `find_dates(...)`.
- New typed API (`extract`) supports absolute, relative, and duration matches.
- Rust kernel is integrated into the `datefinder` package and powers v2/default extraction.
- Added broad conformance, ambiguity, and benchmark tooling for release gating.

## Entry Points

- `datefinder.find_dates(...)`: default v2 compatibility engine.
- `datefinder.find_dates_legacy(...)`: original regex + dateutil engine.
- `datefinder.find_dates_compat(...)`: direct v2 datetime compatibility generator.
- `datefinder.extract(...)`: typed `Match` objects for richer semantics.

## Behavior and Compatibility Notes

- Some ambiguous cases intentionally changed where v2 judgments were semantically stronger.
- For strict legacy parity checks, use `find_dates_legacy(...)`.
- Engine override is available with `find_dates(..., engine="legacy")` or `DATEFINDER_DEFAULT_ENGINE=legacy`.

## Portability Notes

- Compiled Rust artifacts are platform-specific; use prebuilt wheels per OS/arch/Python ABI.
- Release wheel matrix covers:
  - Linux manylinux2014 (`x86_64`, `aarch64`)
  - Linux musllinux_1_2 (`x86_64`, `aarch64`)
  - macOS (`x86_64`, `arm64`)
  - Windows (`x86_64`)
- If no wheel matches, `pip` builds from source and requires Rust tooling.
- `pyo3` `abi3` is enabled to reduce wheel count across Python 3.9+.
