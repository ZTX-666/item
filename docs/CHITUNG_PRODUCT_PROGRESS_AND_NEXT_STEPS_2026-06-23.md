## 1. 客观结论

赤瞳目前已经不是概念阶段，而是一个具备演示能力的本地优先安全智能平台雏形。它已经有前端五大板块、Center 中台、AgentToolbox 工具层、RAG、视觉巡检、WhatsApp 本地库、DocMate 文档能力、外部风险简报、待确认事项和部分工作流能力。

但它还没有达到产品文档里描述的完整形态。文档目标是：

- `4+1` 多智能体完整协同。
- `32` 个 Skill 业务规则包。
- `24` 个固定 Workflow 模板。
- 飞书自然语言入口和确认卡片闭环。
- 视觉自适应巡检闭环。
- 地盘安全主任日常管理的完整闭环。

当前代码实际状态更接近“可演示 MVP + 多模块原型”，不是“可长期稳定交付的生产系统”。最重要的下一步不是继续堆 UI，而是把已经分散存在的工具、页面、API 和工作流固化成 3 到 5 条真正可跑通、可复查、可审计的业务闭环。

## 3. 文档与当前代码的主要差异

### 3.1 文档中已过时的地方

部分文档写的是“已具备 2 个 Skill”，但当前代码里已经有约 8 个 Skill：

- `daily-risk-briefing`
- `hazard-intake`
- `knowledge-query`
- `visual-patrol`
- `docmate-edit`
- `shanshan-doc`
- `whatsapp-sql-query`
- `whatsapp-wacli-ops`

飞书文档里列为“还需开发”的部分能力，当前 AgentToolbox 已经有基础实现：

- `feishu_parse_message_event`
- `feishu_parse_card_action`
- `feishu_event_to_platform_message`
- `feishu_build_center_route_payload`
- `feishu_send_interactive_card`
- `feishu_build_safety_review_card`

这说明项目实际进度比部分早期规划文档更靠前。

### 3.2 文档目标仍未达到的地方

文档里的 32 Skill、24 Workflow、视觉自适应策略包、飞书五阶段完整生态，并没有全部实现。当前更多是：

- 工具分散存在。
- 页面分别可用。
- 中台可以路由部分 intent。
- 某些核心流程可以演示。
- 但业务闭环还没有统一固化。

最典型的缺口是：

- “发现隐患 -> 取证 -> 查制度 -> 生成整改通知 -> 发送确认卡 -> 分判商反馈 -> 视觉复查 -> 关闭隐患”还不是一条稳定的端到端产品流程。
- “飞书消息 -> Center Chat -> Workflow -> 飞书确认卡 -> 确认后执行 -> 更新原卡片状态”还需要做真实回调闭环。
- “视觉自适应巡检策略包”目前有概念和部分视觉能力，但策略包、误报反馈、自动调参、巡检计划还未产品化。

## 4. 当前最有价值的功能方向

### 4.1 第一优先级：把 3 条核心闭环做扎实

不要先追求 24 个 Workflow 全部铺开。建议先做 3 条能代表产品价值的闭环。

#### 闭环 A：隐患发现与整改闭环

目标流程：

1. 安全主任输入“B2 区临边护栏被拆，通知分判商整改”。
2. Center 识别为 `hazard_intake`。
3. 系统创建候选隐患。
4. 关联 CCTV/图片/聊天证据。
5. RAG 查询制度依据。
6. 闪闪文档生成整改通知草稿。
7. 飞书或本地确认卡人工确认。
8. 确认后发送通知。
9. 分判商回传整改照片。
10. 视觉复查。
11. 安全主任确认关闭。

这条闭环最能体现“望风险、闻动态、问制度、成文稿、统全局”。

当前基础：

- `hazard-intake` Skill 已有。
- `workflow_hazard_intake` 已有雏形。
- pending confirmations 已有。
- VLM/视觉、RAG、DocMate、WhatsApp/飞书工具都有基础。

主要缺口：

- 端到端状态机不完整。
- 整改通知与确认卡还未完全绑定。
- 分判商反馈读取和复查流程还薄。

#### 闭环 B：每日开工风险简报

目标流程：

1. 每天早上定时触发。
2. 抓取香港天气和外部安全动态。
3. 汇总未关闭隐患、高风险事项、证书到期、今日巡检重点。
4. 生成早会安全提示和检查任务。
5. 安全主任确认后推送。

当前基础：

- 外部风险页面已打通。
- `workflow_daily_risk_briefing` 已有。
- `fetch_hko_weather`、`fetch_hk_safety_updates`、`draft_daily_risk_briefing` 已有。
- 外部风险历史入库已做。

主要缺口：

- 简报还没有稳定纳入内部隐患、证书、巡检计划。
- 定时调度与推送闭环不完整。
- 飞书发送后的状态更新还不足。

#### 闭环 C：制度问答与智能填表

目标流程：

1. 用户问“这个吊运隐患要填什么表？”
2. RAG 查询制度依据。
3. 搜索相关表格模板。
4. 选择模板。
5. 从隐患或描述中预填字段。
6. 人工确认字段。
7. 导出 DOCX 或记录入库。

当前基础：

- RAG 已可上传和问答，并修复了 OCR 和元数据稳定性。
- SmartForm、TableMapping、DocMate 页面已存在。
- `workflow_form_filling` 有模板。
- AgentToolbox `forms.py`、DocMate DOCX 工具有基础。

主要缺口：

- “制度问答 -> 表格推荐 -> 字段预填 -> 人审 -> 导出”的前后端衔接还不顺。
- 表格模板真实质量和字段映射需要整理。
- 输出文件的人审体验需要强化。

## 5. 模块级下一步建议

### 5.1 赤瞳守护者：视觉巡检

当前状态：

- 有 CCTV 参数接入。
- 有视觉巡检页面。
- 有 YOLO/VLM fallback。
- 有视觉检测历史。
- 有 RAG 安全制度润色提示词。

下一步：

- 建立“巡检策略包”数据结构：`priority_cameras`、`priority_checks`、`vlm_review_prompt`、`thresholds`、`linked_workflow`。
- 把视觉检测结果统一写入 `safety_cases` 或候选隐患，而不是只停留在检测报告。
- 增加整改前后图片对比和复查建议。
- 对每次误报/真隐患做反馈记录，用于下一轮策略调整。

### 5.2 赤瞳聆讯：WhatsApp / 通讯

当前状态：

- wacli 同步、SQLite 查询、分页、媒体预览、聊天列表已做。
- SQL 查询和命令工具已可用。
- WhatsApp 页面已较成熟。

下一步：

- 做“聊天记录隐患挖掘”：从群消息里抽取候选隐患。
- 将 WhatsApp 附件、图片、PDF 和消息上下文挂到安全案例。
- 加入“从聊天生成整改通知草稿”的快捷工作流。
- 对接飞书确认卡，统一通讯确认体验。

### 5.3 耀耀慧读：RAG / OCR / 外部风险

当前状态：

- RAG 知识库有内置 safety 集合。
- PDF OCR 兜底已补。

- 外部风险页面和历史简报已可用。
- 结构化输入页面已有。

下一步：

- 提升 RAG 引用质量：回答必须展示来源文件、页码或 chunk。

- 建立制度版本和集合管理。
- 外部风险简报要和内部隐患、证书、巡检计划联动。

### 5.4 闪闪文档：文书和表格

当前状态：

- 文档审阅、DocMate 改稿、智能填表、表格映射、报告生成都有前端入口。
- Center 和 AgentToolbox 已有 DOCX、forms、report 工具链。

### 5.5 赤瞳中台：编排和确认

下一步：

## 6. 建议 4 周路线图

### Week 1：统一工作流和确认底座

目标：不要再让各页面各跑各的，要有统一业务闭环。

任务：

- 梳理并固定 3 条核心 workflow：隐患整改、每日风险简报、制度问答填表。
- 为每条 workflow 定义固定输入、步骤、产物、确认点、数据库写入。
- 做 workflow run 详情页或至少 API。
- 确认 `pending_confirmations` 与实际动作执行的映射。

交付物：

- `workflow_hazard_rectification`
- `workflow_daily_start_briefing`
- `workflow_policy_form_filling`
- 待确认动作可以从前端确认后真正执行后续动作。

### Week 2：飞书 MVP 闭环

目标：飞书成为移动端入口和确认入口。

任务：

- 配置飞书事件回调。
- 建立用户/群组绑定。
- 飞书消息转 Center chat。
- Center 回复转飞书文本/卡片。
- 卡片按钮回调解析并更新 confirmation。

交付物：

- 飞书里输入“生成今日风险简报”可以得到简报草稿。
- 飞书卡片点击确认后，状态写回本地。

### Week 3：视觉整改闭环

目标：视觉能力不只是检测，而是进入隐患闭环。

任务：

- 检测结果统一生成候选 safety case。
- 加入视觉证据与案例绑定。
- 做整改前后图片复查。
- 建立误报/真隐患反馈表。
- 将视觉巡检接到隐患整改 workflow。

交付物：

- 从 CCTV/图片检测到隐患，可以生成整改通知并复查关闭。

### Week 4：文书和 RAG 产品化

目标：让“问制度”和“成文稿”真正服务安全主任。

任务：

- 整理高频文书模板：整改通知、警告信、停工令、复工通知、事故报告、日报。
- RAG 回答显示引用来源和 OCR 质量。
- 表格填报从模板推荐到 DOCX 输出跑通。
- 对报告生成接入真实隐患、外部风险、证书和巡检数据。

交付物：

- 可演示：问制度 -> 推荐表格 -> 预填 -> 人审 -> 导出。
- 可演示：隐患闭环自动形成报告摘要。

## 7. 暂时不建议优先做的事

以下功能有价值，但不建议现在优先做：

- 一次性补齐 32 个 Skill。
- 一次性实现 24 个 Workflow。
- 大规模重构前端 UI。
- 引入复杂 LangGraph，如果当前自研 workflow engine 足够支撑 MVP。
- 做太多静态展示页，而不是闭环功能。
- 接太多外部系统，导致核心流程不稳定。

原因是当前最缺的是“稳定闭环”，不是“更多入口”。

## 9. 最终判断

赤瞳当前已经具备“高级演示版 MVP”的基础，特别是以下能力已经比较有说服力：

- 五大板块框架。
- 本地优先架构。
- AgentToolbox 工具层。
- RAG 和 OCR。
- WhatsApp 本地数据。
- 视觉巡检。
- DocMate 文档能力。
- 外部风险简报。
- 待确认动作。

但如果要成为真正可交付产品，下一阶段必须把“工具集合”升级成“稳定业务闭环”。建议不要再继续横向铺很多页面，而是集中做 3 条端到端流程：

1. 隐患整改闭环。
2. 每日风险简报闭环。
3. 制度问答与表格填报闭环。

这 3 条跑通后，赤瞳的产品价值会从“功能很多”变成“安全主任真的可以每天用”。
