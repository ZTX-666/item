# 文稿修改助手 — 产品优化建议报告

> 基于领导"人机协同、界面越简单越好"理念，对 publish4 现有产品的系统性诊断与优化建议

---

## 一、核心诊断：产品偏离了领导的核心诉求

领导的原话是：

> "界面越简单越好，计划做一个文稿修改软件，粘贴进来一段文字，旁边有豆包浮窗，语音说修改要求，AI 输出修改文，问使用者是否可行、是否替换。使用者同意就替换上去。"

**一句话概括领导想要的**：写字板 + 语音改稿 + 确认替换，三个动作。

**当前实现的问题**：做了6种AI模式（修改/润色/问答/风险/口语/表格），界面像 VS Code（侧栏+文件树+编辑器+AI面板），用户第一眼看到的是一个"IDE"而不是"写字板"。

这是典型的 **功能蔓延**——开发团队有热情做更多功能，但偏离了"极简"这个核心定位。

---

## 二、P0 级优化（必须修，否则产品不可用）

### 2.1 语音识别：Web Speech API 在中国大陆不可用

**现状**：使用 `window.webkitSpeechRecognition`（Chrome Web Speech API）
**问题**：
- 该 API 后端走 Google 服务器，中国大陆需要翻墙
- 仅 Chrome 支持，Edge/Firefox 不稳定
- 这直接导致领导要求的"语音说"功能在实际使用中无法工作

**建议**：
| 方案 | 延迟 | 中文效果 | 费用 | 推荐度 |
|------|------|---------|------|--------|
| **Whisper 本地推理** | ~1秒（tiny模型） | ★★★★★ | **免费** | **首选**（零成本、离线可用） |
| Vosk 离线引擎 | <0.5秒 | ★★★☆☆ | 免费 | 备选（更轻量但中文弱） |
| Windows SAPI | 快 | ★★☆☆☆ | 免费 | 降级备选（中文识别差） |

**首选方案：Whisper 本地**
- OpenAI 开源模型，在本地 CPU 上运行，**不联网、不花钱**
- 中文模型：`tiny`（75MB，CER约7-8%）或 `base`（142MB，CER约5-6%），接近商用级
- 集成方式：Electron 主进程调 `whisper.cpp` 预编译二进制，通过子进程通信
- 建筑行业术语（混凝土、模板、钢筋等）识别无问题
- 唯一代价：首次下载模型约 75-142MB，之后完全离线

**具体做法**：
1. 麦克风录音存临时 WAV → 2. 喂给 whisper-cli 子进程 → 3. 拿到文字 → 4. 发给豆包 API
整个链路只有豆包 API 花钱，语音转写 = 0 成本

### 2.2 模式精简：6 种模式 → 智能路由

**现状**：修改/润色/问答/风险/口语/表格 6个独立模式，用户需要先选模式再输入
**问题**：
- 违背"界面越简单越好"
- 用户需要理解6种模式的区别，认知负担重
- 领导的描述里只有一种交互："说一句话，AI改"

**建议**：取消手动模式切换，改为 **Intent Router（意图路由）**：

```
用户输入 → LLM 意图识别 → 自动分发到对应处理逻辑
  ├ "把第一段改正式" → 修改模式
  ├ "全文润色为正式语气" → 润色模式
  ├ "这段话有什么风险" → 风险扫描
  ├ "这段口语转成公文" → 口语转正式
  └ "把这段文字做成表格" → 表格生成
```

**好处**：
- 界面只保留一个输入框 + 麦克风按钮
- 用户不需要知道有多少模式，只需"说话"
- 问答模式保留为独立入口（因为它是"问"而不是"改"）

### 2.3 文本替换可靠性

**现状**：`EditorPanel.vue` 的 `replaceText` 方法使用 `node.text.indexOf(oldText)` 做精确匹配
**问题**：
- LLM 返回的 `old_text` 经常与原文有小差异（多余空格、标点半角/全角、换行差异）
- 替换失败时只返回 false，没有给用户任何反馈
- 代码里的 fallback 逻辑（逐字符计算位置）也容易出错

**建议**：
1. **后端校验**：在 `callLlmRevision` 中，如果 `old_text` 不在原文中，尝试模糊匹配（去除首尾空白、统一全角半角标点后再匹配）
2. **前端多重尝试**：替换失败时，依次尝试——精确匹配 → trim 匹配 → 标点归一化匹配 → diff 算法匹配
3. **失败兜底**：所有匹配都失败时，不静默失败，而是在 Diff 区域高亮提示"AI 定位的文字与原文有偏差"，让用户手动确认

### 2.4 Diff 可视化应移入编辑器

**现状**：Diff（红色删除线旧文 + 绿色新文）只在右侧 AI 面板的小气泡中展示
**问题**：
- 用户无法在上下文中看到修改效果
- 领导说"问使用者是否可行"，但当前只能看到脱节的两个片段

**建议**：在编辑器中直接标注修改建议（使用 Tiptap 自定义 Mark 扩展）：
- 原文保持，被修改部分加上 `background: #fef2f2` + 删除线
- 新文以 `background: #f0fdf4` 插入在旁边
- 采纳后自动替换，拒绝后恢复原样
- 这才是真正的"人机协同"视觉体验

### 2.5 撤销 AI 修改

**现状**：采纳替换后，只能用 Ctrl+Z 全局撤销，没有针对性的 AI 修改回退
**问题**：如果用户采纳了3处 AI 修改后想回退第1处，只能逐个撤销后面的操作

**建议**：维护一个 `revisionHistory` 栈，每次 AI 修改记录 `{oldText, newText, position}`，提供"撤销上一条AI修改"的专用按钮。

### 2.6 首屏体验：去掉侧栏，一开就是写字板

**现状**：打开后是 VS Code 布局——左侧文件树 + 中间编辑器 + 右侧 AI 面板
**问题**：领导说"界面越简单越好，就像写字板"，但当前界面更像个 IDE

**建议**：
- **默认首屏**：全屏写字板 + 底部/右侧可收起的 AI 浮窗
- 文件树默认折叠，通过按钮展开
- 编辑区工具栏只保留：撤销/重做 + "AI改这段"
- 打开即粘贴即用，不需要先建文件

### 2.7 流式输出

**现状**：AI 思考时只显示"思考中..."，等全部返回后才一次性展示
**问题**：等待体验差，大模型回复可能需要5-10秒，用户以为卡死了

**建议**：QA 模式已有流式支持（`ai-services.cjs` 的 `stream: true`），但修改/润色模式没有。建议所有模式都支持流式，先用打字机效果显示"AI正在思考修改方案..."，然后流式展示 Diff 结果。

## 三、P1 级优化（应该做，体验完整性）

### 3.1 .docx 导入/导出

**现状**：代码中有 `.docx` 的占位提示（"请安装 mammoth"），实际未实现
**问题**：目标用户（中国建筑国际集团）的公文几乎都是 .docx 格式，不支持 .docx 等于无法接入真实工作流

**建议**：
- 导入：集成 `mammoth.js`（.docx → HTML），已有的 `readFileContent` 预留了扩展点
- 导出：集成 `docx` npm 包（HTML → .docx），或 `html-docx-js`
- 优先级极高，这是用户能不能真正用起来的关键门槛

### 3.2 修改历史时间线

**现状**：每次 AI 修改独立，无法回溯"刚才 AI 帮我改了什么"
**建议**：
- 右侧面板增加"修改历史"标签页
- 时间线展示：`14:32 "精简第一段" → 采纳` / `14:35 "正式化结尾" → 拒绝`
- 点击历史条目可在编辑器中高亮对应位置

### 3.3 连续对话上下文

**现状**：每次修改指令独立发送，AI 不知道之前的对话
**问题**：用户可能说"还是不太对，再改改"——AI 不知道"再改改"指什么

**建议**：QA 模式已有 `qaHistory`，将同样的机制扩展到修改/润色模式，保留最近 6 轮对话上下文。

### 3.4 快捷指令

**现状**：润色模式有5个语气按钮（专业正式/亲切友好/简洁直白/自信有力/客观中立），但修改模式没有快捷操作
**建议**：在工具栏或输入框旁增加常用指令按钮：
- 「正式化」→ "把选中文本改得更正式"
- 「精简」→ "精简选中文本，去掉冗余"
- 「加总结」→ "在选中文本后加一段总结"
- 「展开」→ "展开选中文本的细节"

### 3.5 导出为 PDF

**现状**：不支持
**建议**：集成 `html2pdf.js` 或 Electron 的 `webContents.printToPDF()`

## 四、P2 级优化（锦上添花，差异化竞争力）

### 4.1 3311 管控规则嵌入知识库

当前知识库功能已实现，但需要预置建筑行业 3311 管控的规范模板，让 AI 自动检查文稿是否符合管控要求。

### 4.2 多人协同编辑

集成 Y.js + WebSocket，实现多人实时编辑。但这是 Phase 4 级别的需求，当前不建议优先。

### 4.3 模板库

预置公文/方案/汇报/总结等模板，用户选模板后填空。

### 4.4 修改审计日志

记录"谁在什么时间修改了什么"，用于合规审计。

### 4.5 批量修改指令

一次下达多个修改指令，AI 逐条处理。

### 4.6 品牌融入

将"闪闪/耀耀"吉祥物和"耀文"品牌名融入界面，增强归属感。

### 4.7 离线语音识别

当前默认使用 Whisper 本地模型做语音识别，不依赖云服务。如需进一步优化可考虑 GPU 加速推理。

## 五、架构层面的建议

### 5.1 前端瘦身

当前 `AiPanel.vue` 有 547 行代码，承担了太多职责（6种模式处理 + 消息管理 + 语音 + 提案管理）。建议拆分：
- `useChat.ts` — 消息管理
- `useRevision.ts` — 修改/润色/口语提案逻辑
- `useRiskScan.ts` — 风险扫描
- `useQA.ts` — 问答逻辑
- AiPanel 只负责 UI 渲染

### 5.2 Prompt 可维护性

当前 Prompt 散落在 `llm.cjs` 和 `ai-services.cjs` 两个文件中，建议：
- 统一放在 `prompts/` 目录下
- 每种任务一个 `.md` 文件
- 支持热更新（不需要重新打包）

### 5.3 错误处理

当前 AI 调用失败只显示"处理失败：xxx"，建议：
- 区分网络错误 / API Key 错误 / 返回格式错误 / 文本匹配失败
- 给出针对性修复建议
- 网络错误时提供"重试"按钮

## 六、总结

| 优先级 | 核心问题 | 影响 | 工作量 |
|--------|---------|------|--------|
| P0 | 语音识别不可用→Whisper本地 | 核心功能不能用 | 2天 |
| P0 | 6模式→智能路由 | 违背"极简"定位 | 3天 |
| P0 | 替换不可靠 | 采纳后替换失败 | 2天 |
| P0 | Diff不在编辑器内 | 体验割裂 | 2天 |
| P0 | 首屏不是写字板 | 第一印象偏离 | 1天 |
| P1 | 不支持 .docx | 无法接入真实工作流 | 3天 |
| P1 | 无修改历史 | 无法回溯 | 2天 |
| P1 | 无连续对话 | 交互不自然 | 1天 |

**一句话总结**：产品功能做多了，核心工作流反而没打通。先把"语音说 → AI改 → 确认替换"这条线做到极致流畅，再考虑扩展其他能力。
